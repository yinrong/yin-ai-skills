# 设计理念

## 核心问题

AI 辅助开发面临的关键挑战：

1. **过程不透明**：AI 做探索性任务时，人类难以理解思路
2. **经验不积累**：实验失败后重来，之前的尝试丢失
3. **无法接力**：AI 会话中断后，新会话无法继续
4. **难以复现**：记录过于简略，跳跃性强

## 解决方案

### 1. 记录 > 记忆

**原则**：不依赖上下文，所有信息持久化到文件。

**实现**：
- 每个决策点记录理由
- 每次失败记录原因
- 每个检查点保存代码快照
- 所有输出结构化存储

**好处**：
- 新会话可以无缝接手
- 人类可以随时查看进度
- 团队可以协作和复现

### 2. 失败 = 认知

**原则**：失败路径和成功路径同等重要。

**实现**：
- `失败案例库.md` 记录所有失败
- 每个失败标注原因和教训
- 失败的代码保留在 `failed_attempts/`
- 决策树标注"死胡同"分支

**好处**：
- 避免重复踩坑
- 理解问题的边界
- 学习调试思路

### 3. 人类可读

**原则**：非技术人员也能看懂思路。

**实现**：
- `README.md` 索引文件（5 分钟快速了解）
- 用表格、列表组织信息
- 决策理由用自然语言描述
- 避免大段代码，只保留关键部分

**好处**：
- PM/设计师可以参与
- 新成员快速上手
- 便于向上汇报

### 4. AI 可读

**原则**：新 AI 会话能通过读取记录继续工作。

**实现**：
- 固定的文件结构和命名
- 明确的"下一步"指引
- 完整的上下文信息
- 可执行的复现脚本

**好处**：
- 断点续传无损
- 不同 AI 可以接力
- 实验可以自动化

### 5. 中文优先

**原则**：文件名、内容全部中文，符合国内习惯。

**实现**：
- 目录名：`调研/`、`实验/`、`检查点/`
- 文件名：`状态.md`、`发现.md`、`复现指南.md`
- 内容：中文描述，英文术语保留

**好处**：
- 更直观、更快速
- 降低认知负担
- 符合团队习惯

## 设计决策

### 为什么用 Markdown？

- ✅ 人类可读（纯文本）
- ✅ Git 友好（diff 清晰）
- ✅ 工具支持广泛（GitHub、VS Code、Obsidian）
- ✅ 可渲染为网页

### 为什么分多个文件？

- ✅ 模块化（按主题组织）
- ✅ 减少冲突（团队协作）
- ✅ 按需加载（只读相关部分）
- ✅ 便于归档（删除不需要的部分）

### 为什么保留失败尝试？

- ✅ 避免重复错误
- ✅ 理解问题边界
- ✅ 学习调试方法
- ✅ 展示思考过程

### 为什么要索引文件？

- ✅ 快速了解全局（5 分钟）
- ✅ 降低阅读成本
- ✅ 方便跳转导航
- ✅ 适合汇报演示

## 对比其他方案

### vs. 纯 Git Commit

| | Git Commit | yin-ai-skills |
|---|---|---|
| 记录粒度 | 粗（每次提交） | 细（每轮迭代） |
| 失败记录 | ❌ | ✅ |
| 决策理由 | 简略（commit message） | 详细（独立文件） |
| 人类可读 | 需要 `git log` | 直接读 README |
| AI 可读 | 困难 | 优化过的结构 |

### vs. Jupyter Notebook

| | Jupyter | yin-ai-skills |
|---|---|---|
| 适用场景 | 数据分析 | 通用开发 |
| 版本控制 | 差（JSON 格式） | 好（Markdown） |
| 团队协作 | 冲突多 | 冲突少（多文件） |
| 断点续传 | ✅ | ✅ |
| 失败记录 | 手动 | 自动 |

### vs. 项目文档

| | 传统文档 | yin-ai-skills |
|---|---|---|
| 更新频率 | 低（事后补） | 高（实时生成） |
| 完整性 | 差（选择性记录） | 好（强制模板） |
| 过程记录 | ❌ | ✅ |
| 可执行性 | ❌ | ✅（复现脚本） |

## 适用场景

### 适合

- ✅ 探索性任务（不知道答案）
- ✅ 多轮迭代优化
- ✅ 需要团队协作
- ✅ 需要长期维护
- ✅ 需要复现验证

### 不适合

- ❌ 一次性脚本（overhead 太大）
- ❌ 简单修改（如改个变量名）
- ❌ 紧急 hotfix（来不及记录）
- ❌ 个人临时实验（不需要复现）

### 边界情况

**问**：多少轮迭代值得用这个工具？

**答**：≥5 轮。少于 5 轮的任务，手动记录更快。

**问**：实验失败了怎么办？

**答**：记录失败原因，标记为"未完成"，归档到 Git。失败也是认知。

**问**：记录太多会不会影响 AI 性能？

**答**：不会。AI 按需读取文件，不会一次性加载全部。索引文件控制在 200 行以内。

## 反模式

### 不要做的事

❌ **过度记录**：不要记录每个 print 语句的输出

❌ **复制粘贴代码**：用 Git commit + 引用，不要全文复制

❌ **忽略索引**：README.md 必须保持最新

❌ **跳过失败记录**：失败也要记录，不要只记录成功

❌ **手动维护**：让 AI 自动生成，不要人工编辑太多

### 要做的事

✅ **保持简洁**：只记录关键决策和结果

✅ **及时更新**：每轮迭代后立即更新状态

✅ **结构化**：使用表格、列表，不要大段文字

✅ **可执行**：提供 `复现指南.sh`，不要只有文字描述

✅ **版本控制**：及时 commit，不要积累太多再提交

## 总结

yin-ai-skills 的设计目标是让 AI 辅助开发：

1. **可追溯**：每个决策都有记录
2. **可复现**：任何人都能重做
3. **可协作**：团队可以接力
4. **可学习**：失败经验也保留

核心是**用结构化的记录替代依赖记忆**，让知识持久化。
