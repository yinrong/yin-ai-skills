# DDD + TDD 开发流程

文档驱动设计（DDD）+ 测试驱动开发（TDD），以 E2E 测试为核心验证手段。

## 核心原则

1. **文档先行（DDD）**：先确认"做什么"和"怎么设计"，再动手
2. **E2E 测试先行（TDD）**：先写端到端测试（应 FAIL）→ 再写实现（直到 PASS）
3. **禁止 mock / 单元测试优先**：mock 掩盖真实问题，单元测试增加代码量但不验证真实行为
4. **最小代码量**：测试代码也是代码，能省则省

## 流程顺序（不可颠倒）

```
1. DDD: 更新文档（FEATURES.md / CLAUDE.md）
   ↓
2. TDD: 写 E2E 测试（连接真实服务，验证完整链路）
   ↓  测试应 FAIL
3. 实现代码（直到测试 PASS）
   ↓
4. 重构 → 提交
```

## 测试策略

### 必须做的：E2E 测试

- 连接真实服务，走完整链路
- 验证用户可见的行为和结果
- 覆盖正常路径 + 关键异常路径

### 原则上省略的

| 类型 | 态度 | 原因 |
|------|------|------|
| Mock 测试 | **禁止** | 掩盖真实问题（如认证、网络、状态），给虚假安全感 |
| 单元测试 | **省略** | 增加代码量，不验证真实集成行为，重构时成为负担 |
| 集成测试（部分 mock） | **省略** | 半真半假，不如全真 |

### 什么时候可以写单元测试（例外）

仅当以下条件**全部满足**时：
- 纯计算逻辑（无 IO、无状态、无副作用）
- 逻辑复杂度高（分支多、边界多）
- E2E 测试无法覆盖该逻辑的所有分支

例如：复杂的价格计算函数、解析器、编码转换。

## 文档规范（DDD 部分）

### 三文件各有受众，不重复

| 文件 | 受众 | 内容 |
|------|------|------|
| `CLAUDE.md` | AI | 架构约束、开发规范、关键决策 |
| `README.md` | 人 | 是什么、怎么装、怎么跑 |
| `FEATURES.md` | AI+人 | 功能清单 + 状态 + 测试映射 |

### 不需要的文件

- ❌ `ARCHITECTURE.md` — 与 CLAUDE.md 重复
- ❌ `CHANGELOG.md` — git log 已有
- ❌ `TODO.md` — 与 FEATURES.md 重复

## E2E 测试编写指南

### 原则

1. **测真实环境**：连真实数据库、真实 API、真实服务
2. **测用户行为**：模拟用户的完整操作流程
3. **最小断言**：只验证用户可见的结果，不验证内部实现细节
4. **独立可跑**：每个测试自包含，不依赖其他测试的执行顺序

### 目录结构

```
tests/
├── e2e/           # 端到端测试（必须）
│   ├── smoke.test.ts      # 冒烟测试：核心功能可用
│   ├── <feature>.test.ts  # 按功能分文件
│   └── setup.ts           # 环境准备（真实服务连接）
└── unit/          # 仅在例外情况使用（纯计算逻辑）
```

### 前端 E2E

- 用 Playwright 模拟浏览器操作
- 穷举常见情况：正常输入、空输入、特殊字符、网络断开
- 不能只验证 API，必须验证 UI 行为

### 后端 E2E

- 启动真实服务实例
- 通过 HTTP/CLI 调用，验证响应
- 数据库用真实实例（可以是测试库，但不是 mock）

## 反模式（禁止）

```
❌ 先写代码再补文档/测试
❌ 用 mock 服务替代真实服务测试
❌ 写大量单元测试来提高"覆盖率"
❌ 为了测试而引入依赖注入/接口抽象（过度设计）
❌ 测试内部实现细节（如：某个私有方法被调用了几次）
```

## 使用方法

开发新功能时遵循此流程：

```bash
# 1. 更新功能文档
# 编辑 FEATURES.md，添加新功能描述和验收标准

# 2. 写 E2E 测试（应失败）
# 创建/修改 tests/e2e/<feature>.test.ts

# 3. 运行测试确认 FAIL
npm test  # 或项目对应的测试命令

# 4. 实现功能代码（直到 PASS）

# 5. 重构 → 提交
```
